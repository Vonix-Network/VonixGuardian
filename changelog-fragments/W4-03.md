### W4-03 — Fabric event wiring parity (native API only)

Bring Fabric cells (all four MC versions) closer to Forge-equivalent event
coverage using only events natively exposed by the Fabric API — no mixins.
Companion work items W4-04 (block state changes) and W4-06 (container + item
ops) will land the mixin-backed handlers.

**Added on 1.19.2 / 1.20.1 / 1.21.1 fabric cells:**
- `AttackEntityCallback` → HANGING_BREAK (player-caused item-frame / painting
  destruction, mirroring Forge `AttackEntityEvent`).
- `ServerEntityEvents.ENTITY_LOAD` now fans out `HangingEntity` instances to a
  HANGING_PLACE submit path (matching Forge `EntityJoinLevelEvent` handling for
  frames / paintings / leash knots).
- `AttackBlockCallback` now runs the same async `Guardian#lookupAtPos` on
  inspecting players that Forge's `PlayerInteractEvent.LeftClickBlock` runs,
  via a shared daemon worker (`VonixGuardian-FabricEvents-Worker`).

**Added on 1.18.2 fabric cell:** same three, plus the async lookup uses
`sendMessage(Component, UUID)` since 1.18.2 has no `sendSystemMessage`.

**Explicit Fabric-API gaps** are now enumerated in a single
`TODO(v1.2.0): mixin — W4-04/W4-06` block at the top of each `register()`
method: block place, universal mob griefing, explosion detonate, piston pre,
container open/close, bucket fill/empty, item drop/pickup/craft, sign change,
fire burn/ignite, ice/snow fade, dispense, leaves decay, block form/spread.
Nothing is "kind of" wired — the audit trail is honest about what needs a
mixin. 1.18.2 additionally flags CHAT / COMMAND / DAMAGE_HISTORY / natural
AFTER_DEATH as mixin-only because that Fabric API predates
`ServerMessageEvents` and `ServerLivingEntityEvents`.
